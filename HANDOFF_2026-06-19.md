# AlbumPlace V1 Release Handoff - 2026-06-19

## Critical Date/Recent-File Fix

### Symptom

- When old photos were copied into generated location albums such as `Pictures/금산에서/`, KakaoTalk and Instagram showed those copied photos in their "recent photos" picker.
- Instagram was fixed after setting MediaStore dates and file `lastModified`, but KakaoTalk still showed the copied photos until media scanner refresh was restored.

### Root Cause

- The source/original photo date was not necessarily being changed.
- The problematic item was usually the newly created album copy.
- KakaoTalk recent picker is sensitive to filesystem/media metadata on the copied file:
  - actual file `mtime`
  - MediaStore `date_modified`
  - media scanner refreshed state
- During the P0 source-date safety cleanup, extra post-copy updates were reduced. That protected originals, but it also weakened the previously fixed KakaoTalk recent-file defense.

### Important Distinction

- Original photos must not be moved, renamed, overwritten, or have their MediaStore rows updated.
- Generated album copies may need their own metadata corrected so they do not appear as newly taken/recent photos in other apps.
- Videos intentionally use move behavior when the user option is enabled.

### Current Fix

In `MainActivity.copyMediaItem()` for image copies:

1. Insert copied image into the target location album.
2. Preserve copied row dates using `putMediaDates()`:
   - `DATE_TAKEN = takenAt`
   - `DATE_ADDED = takenAt / 1000`
   - `DATE_MODIFIED = takenAt / 1000`
3. After copy completion, call `setCopiedFileModifiedTime(targetUri, item.takenAt)`.
4. `setCopiedFileModifiedTime()`:
   - resolves the copied file path from MediaStore `_data`
   - falls back to `RELATIVE_PATH + DISPLAY_NAME`
   - calls `File#setLastModified(takenAt)`
   - calls `MediaScannerConnection.scanFile(...)`

This matches the earlier investigation in:

- `reports/media-compare-20260609_233202.md`
- `scripts/repair-gallery-dates.ps1`

### Verified By User

- Instagram recent picker no longer showed the copied old photo.
- KakaoTalk recent picker was fixed after adding `MediaScannerConnection.scanFile(...)`.

### Caution For Future Changes

Do not remove the post-copy file-time restoration and media scan unless replacing the copy strategy entirely.

Specifically keep this chain for image album copies:

```text
MediaStore insert
copy bytes
IS_PENDING = 0
DATE_TAKEN / DATE_ADDED / DATE_MODIFIED set to takenAt
File#setLastModified(takenAt)
MediaScannerConnection.scanFile(copiedPath)
```

If KakaoTalk recent photos regress again, first compare copied file metadata:

```text
file mtime
MediaStore date_modified
MediaStore date_added
MediaStore datetaken
owner_package_name
```

The old comparison showed EXIF and bytes were identical; the issue was file/MediaStore timestamps.

## Device Date Repair - 2026-06-20

- Found 505 image rows in `DCIM/Camera/` re-registered around 2026-06-17 23:41 KST.
- The affected rows had a recent `DATE_ADDED`, which caused old images to appear in recent-photo pickers.
- Backed up the original MediaStore rows to `date-repair-mediastore-backup-20260620-214402.txt`.
- Built the verified repair plan in `date-repair-plan-20260620-214402.csv`.
- Restored `DATE_ADDED` and `DATE_MODIFIED` using verified filename timestamps or an already-correct old `DATE_MODIFIED` value.
- Protected MediaStore columns required: set `IS_PENDING=1`, update dates, then set `IS_PENDING=0`.
- Successfully restored 497 of 505 rows.
- Left 8 rows unchanged because they had no EXIF date, no date in the filename, and no trustworthy old timestamp.
- Do not assign guessed dates to those 8 files.

## Samsung Gallery Direct Album Investigation - 2026-06-20

- Verified that a generated physical album can be opened directly on the connected Samsung device.
- Action: `com.android.gallery.action.SHORTCUT_ALBUM_VIEW`
- Component: `com.sec.android.gallery3d/com.samsung.android.gallery.app.activity.external.GalleryExternalActivity`
- Pass MediaStore `BUCKET_ID` as integer extra `ALBUM_ID`.
- Physical album extras: `key-album-type=0`, `IS_VIRTUAL_ALBUM=false`, `key-show-album-info=false`.
- Verified `Pictures/송파구에서/` opened as `송파구에서`, showing 43 images and 1 video.
- This is a Samsung-private Intent. A normal gallery fallback is required for unsupported versions/devices.
